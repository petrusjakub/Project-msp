# Single-file product page for Manulife Saving Protector (MSP)

A complete single-file HTML product page implementing a 6-tab layout with sticky WhatsApp header, interactive premium calculator, static benefit illustrations, policy terms, and FAQ accordion. The page is self-contained with no external dependencies, targets a Bahasa Indonesia layperson audience, and mirrors the CSS/layout conventions of the existing template (`index (17).html`). Watch for: the calculator uses different annual benefit rates for USD vs IDR (5%/8%/11% vs 15%/17.5%/20%) but the static Manfaat and Ilustrasi tabs only document the IDR rates, creating a content gap for USD customers (confirmed); the calculator builds result HTML via string concatenation of user-derived values without sanitization, which is an XSS vector (confirmed); the `modalFaktor` divisor approach means a non-tahunan premium input is divided by a sub-1 factor to derive the annual equivalent, and the resulting "annual premium" drives all benefit math — if the factors are even slightly wrong the error compounds across 16+ payment years (likely).

## High-level view

The calculator converts any payment mode (monthly, quarterly, semesterly) to an annual equivalent using currency-specific `modalFaktor` values, then applies currency-specific benefit rates and a universal 500% end-of-term benefit. The static content tabs (Manfaat, Ilustrasi) only document IDR rates (15%/17.5%/20%), while the calculator silently applies lower USD rates (5%/8%/11%). A USD customer reading the Manfaat tab would expect 15% annual cash but the calculator would show 5%.

The WhatsApp integration hardcodes a personal phone number and agent name across four locations (banner, floating button, tooltip, and pre-filled message text). This is a maintenance concern if the agent assignment changes.

1. **USD benefit rates undocumented in static content** — The Manfaat tab shows 15%/17.5%/20% rates without noting these are IDR-only. The calculator uses 5%/8%/11% for USD. Either the static content needs a USD row or the calculator rates need verification against the product spec.

If `parseNumber` is ever bypassed or the input handling changes, unescaped content flows into the DOM. Use `textContent` for user-derived values or sanitize the HTML template.
3. **Benefit ratio discrepancy for USD** — The "Ringkasan Perbandingan Manfaat" table shows 118%/138.5%/164% ratios which only hold for IDR rates. A USD 20-year term with 5%/8%/11% rates yields (16*0.11 + 5.0)/5 = 135.2%, not 164%. The table is misleading for USD customers.
4. **Modal factor precision risk** — The `modalFaktor` values (e.g., IDR bulanan = 0.095) determine annual premium via division. A 1% error in the factor propagates linearly into all benefit calculations across 16 years of payments. These should be verified against official rate tables.
5. **Hardcoded contact info in four places** — The agent phone number and name appear in the WA banner href, banner text, floating button href, and tooltip. Consider extracting to a single JS constant to avoid partial updates.

## USD vs IDR rate divergence in the calculator

The static Manfaat tab presents a single benefits table:

```
10 Tahun → 15%    15 Tahun → 17,5%    20 Tahun → 20%
```

These rates are stated without qualification. However, the calculator JavaScript defines:

```javascript
var benefitRates = {
    IDR: { 10: 0.15, 15: 0.175, 20: 0.20 },
    USD: { 10: 0.05, 15: 0.08, 20: 0.11 }
};
```

The USD rates are dramatically lower (5% vs 15% for 10-year, 11% vs 20% for 20-year). This creates two problems: a customer reading the Manfaat tab in USD mode would be misled about expected returns, and the "Ringkasan Perbandingan Manfaat" table showing 118%/138.5%/164% ratios is only valid for IDR. For USD 20-year: `(16 * 0.11 + 5.0) / 5.0 * 100 = 135.2%`, not 164%.

If these different rates are correct per the product spec (which is plausible — different currencies often carry different guarantees), the static content needs a parallel table for USD or an explicit disclaimer that the rates shown are IDR-only.

## XSS surface in calculator output

The `calculateMSP` function builds an HTML string and injects it via `resultsEl.innerHTML`. The user input flows through `parseNumber` (which strips non-digits via `str.replace(/[^\d]/g, '')`), then through `formatCurrency` (which adds currency symbols and thousand separators). The current `parseNumber` implementation makes injection difficult since it strips everything except digits. However, the pattern is fragile: any future change to `parseNumber` or addition of a text field that doesn't go through the same sanitization would create a direct XSS path. The safer pattern would use DOM creation or `textContent` for the numeric values, keeping `innerHTML` only for the structural HTML.

## Annual premium derivation via modalFaktor

When a user selects a non-annual payment mode, the entered premium is converted to an annual equivalent by dividing by the mode factor:

```javascript
var annualPremium = premiumRaw / faktor;
// e.g., IDR bulanan: annualPremium = monthlyPremium / 0.095
```

The factor encodes the loading/discount for non-annual payment — it's not simply 1/12 (0.0833) but 0.095, reflecting a ~14% loading for monthly payments. These factors are the foundation of all downstream benefit calculations. If any factor is incorrect relative to the official rate card, the error multiplies across 5 years of premium payments and up to 16 years of benefit payouts. The IDR and USD factors differ slightly (USD bulanan = 0.090 vs IDR = 0.095), which is another product-specific detail that should be verified against the official "Tabel MSP Premi IDR & USD" document in the repository.

## Accessibility gaps

The tabs lack `aria-selected`, `aria-controls`, and keyboard navigation (arrow keys, Enter/Space). The tab panels lack `aria-labelledby`. Screen reader users cannot navigate between tabs using standard ARIA tab patterns. The FAQ accordion uses a CSS `max-height: 600px` transition that will clip content without scroll if an answer grows beyond that height.
